# vue3project

Vue3 + TypeScript + Element Plus frontend, connects to two Kratos backend services (StudentService and ArticleService) via gRPC-to-HTTP.

## Quick Start

Clone the repo, then open 3 terminals to run the complete stack.

### Step 1: Start StudentService backend

```bash
cd demo1kratos
go run ./cmd/demo1kratos -conf ./configs
```

You should see output like:
```
msg=[HTTP] server listening on: [::]:8001
msg=[gRPC] server listening on: [::]:9001
```

### Step 2: Start ArticleService backend

```bash
cd demo2kratos
go run ./cmd/demo2kratos -conf ./configs
```

You should see:
```
msg=[HTTP] server listening on: [::]:8002
msg=[gRPC] server listening on: [::]:9002
```

Both backends use SQLite with volatile storage. Data is not persistent — restarting the service wipes everything.

### Step 3: Start the frontend

```bash
cd vue3project
npm install
npm run dev
```

You should see:
```
VITE v8.x.x  ready in xxx ms

➜  Local:   http://localhost:5173/
```

Visit `http://localhost:5173/` to see the app.

## Demo Preview

The page has a white card on a soft muted background. At the top is a title "Vue3 + Kratos Demo", below it are two tabs: **StudentService** and **ArticleService**.

### StudentService Tab

A gradient title band shows "StudentService (demo1kratos :8001)".

Below is a white card with a data table. At first the table is blank with a hint "No students yet. Click Create to add one."

Click **Create** → a dialog pops up with a form:
- Name (required, validated on focus change)
- Age (numeric input, default 18)
- Class (not required)

Provide the values and click **Submit**. On success, a green toast "Created: id=1, name=xxx" shows at the top, the dialog closes, and the table refreshes to show the new row.

The table has:
- Borders (outside thick, inside dashed)
- Stripe rows (alternating shades)
- Cells centered
- Action buttons: **Select** (logs details), **Update** (edit dialog), **Delete** (removes row)

Click **Update** → same form dialog pre-filled with current values. Edit and submit.

If you submit with a blank Name, the form shows a red validation message "Name is required" beneath the field — the request is not sent.

If the backend returns a non-OK response (e.g. invalid data), a dialog pops up showing:
- Title with HTTP status code
- Reason (e.g. "BAD_PARAM")
- Message (e.g. "student name is blank")

At the bottom, a collapsible **Logs** section shows time-based operation records like:
```
[14:30:01] Loaded 3 students
[14:30:05] Created: id=4, name=Alice
[14:30:08] Updated: id=4, name=Bob
[14:30:10] Deleted: id=4
```

### ArticleService Tab

Switch to the ArticleService tab. Same layout and interaction, with article fields:
- Title (required)
- Content (textarea)
- Student ID (not required, links to a student)

The table shows articles with a "Student" column that displays the linked student ID, showing "-" if none.

## Mechanism

The frontend uses [protobuf-ts](https://github.com/nicolo-ribaudo/protobuf-ts) generated TypeScript clients. These gRPC clients get converted to HTTP clients via [@yylego/grpc-to-http](https://www.npmjs.com/package/@yylego/grpc-to-http), so the web page sends standard HTTP requests to the Kratos backends.

Vite dev mode routes the requests:
- `/demo1kratos-base/*` → `http://127.0.0.1:8001` (StudentService)
- `/demo2kratos-base/*` → `http://127.0.0.1:8002` (ArticleService)

Since the gRPC-to-HTTP module uses axios, backend faults arrive as AxiosErrors. The `parseError()` function extracts the Kratos response data (code, reason, message) and `showErrorDialog()` shows the details.

## Code Structure

```

│   ├── api/transport.ts           # Two transport configs (demo1, demo2) with routing paths
```

## Scope

This is a demo project focused on frontend-backend integration patterns. It does NOT include cross-service data constraints. Things not handled in this demo:

- No validation that a student exists before creating an article linked to that student
- No cascade deletion — deleting a student does NOT delete linked articles
- No restriction on deleting a student that has linked articles

In a production system, these data constraints must be enforced. This project demonstrates the CRUD + gRPC-to-HTTP communication pattern, not business logic completeness.
